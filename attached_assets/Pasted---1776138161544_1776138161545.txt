يوجد خطأ حي على الموقع المنشور وما زال يتكرر رغم التوجيهات السابقة:

المشكلة:
في بعض التمارين يظهر للمستخدم نص آية معيّن، لكن الخيارات المعروضة تحتوي كلمات ليست موجودة داخل النص الظاهر للآية/المقطع.
هذا مخالف تماما للقاعدة المعتمدة:
All options must come from displayed_passage_text only.

هذا ليس issue بسيطا في الواجهة.
هذا bug في منطق التوليد أو الاسترجاع أو النشر، ويجب إصلاحه فورًا على مستوى:
1) generation
2) API delivery
3) published content backfill

════════════════════════════════════
1) قرار فوري
════════════════════════════════════

اعتبر هذا النوع من الأسئلة blocked in production إلى أن ينجح الفحص.

أي سؤال من هذا النوع إذا فشل فيه الشرط التالي:
every option ∈ displayed_passage_text
=> لا يُعرض للمستخدم إطلاقا

بدلا من ذلك:
- skip invalid question
- fetch next valid question
- log incident
- mark item for review/deactivation

════════════════════════════════════
2) المطلوب الآن: Hotfix server-side فوري
════════════════════════════════════

أريد إضافة validation guard على مستوى API / quiz delivery، وليس فقط عند التوليد.

قبل أن يرسل الخادم أي سؤال للواجهة:
تحقق من التالي:

1. displayed_passage_text exists
2. correct answer exists inside displayed_passage_text
3. every distractor exists inside displayed_passage_text
4. no option is outside displayed_passage_text
5. if context_mode = single_ayah
   then all options must be inside that single displayed ayah text
6. if context_mode = multi_ayah
   then all options must be inside the displayed multi-ayah passage only

إذا فشل أي شرط:
- do not return the question to client
- mark question invalid_delivery_candidate
- log question id + failure reason
- fetch another valid question

هذا يجب أن يكون hotfix فوري على الخادم الحي.

════════════════════════════════════
3) Root cause hunt
════════════════════════════════════

أريدك أن تبحث عن السبب الحقيقي في أحد هذه الأماكن:

A. Generator bug
- هل generator ما زال يبني distractors من pool خارجي لا من displayed passage؟

B. Normalization bug
- هل displayed_passage_text مختلف عن passage المستخدم فعليا لتوليد الخيارات؟

C. Delivery bug
- هل API ترجع options من record مختلف عن displayed passage الحالي؟

D. Migration/backfill bug
- هل بعض الأسئلة القديمة بقيت منشورة بصيغة قديمة ولم يطبق عليها النظام الجديد؟

E. Caching bug
- هل الواجهة أو API cache ما زالت تخدم records قديمة invalid؟

لا أريد تخمينات.
أريد trace واضح من database → generation → API → frontend.

════════════════════════════════════
4) Permanent invariant
════════════════════════════════════

أريد تثبيت invariant صريح في الكود:

For this exercise type, no question is valid unless every visible option is extracted from the exact displayed Quran passage shown to the learner.

طبّق هذا invariant في:
- generator
- admin save/update
- publish pipeline
- delivery API
- review tools
- backfill scripts

════════════════════════════════════
5) Backfill on live published data
════════════════════════════════════

أريد script فوري يفحص كل الأسئلة المنشورة من هذا النوع في قاعدة البيانات:

For each published item:
- parse displayed_passage_text
- verify every option exists in displayed_passage_text
- verify correct answer exists in displayed_passage_text

النتائج:
1. valid → keep
2. invalid but repairable via multi_ayah rebuild → rebuild and revalidate
3. invalid and not repairable → deactivate immediately + send to review queue

أريد هذا على المحتوى المنشور الحالي، لا على المحتوى الجديد فقط.
