يرجى تنفيذ تحديث عاجل في quranesh.com:

المطلوب:
إعطاء جميع المستخدمين الحاليين المسجلين قبل تحديث نظام الدفع اشتراكا مجانيا مفتوحا مع وصول كامل للمحتوى، بدون أي مطالبة بالدفع.

الهدف:
أي مستخدم سجل حسابه قبل إطلاق نظام الدفع الجديد يجب اعتباره “grandfathered user” ويحصل على اشتراك مجاني مفتوح تلقائيا.

التنفيذ المطلوب:

1) تحديد المستخدمين المستهدفين
- اعتبر “المستخدمين الحاليين” هم كل من كان لديهم حساب قبل وقت نشر تحديث الدفع الجديد.
- استخدم cutoff واضحا من created_at أو تاريخ نشر migration.
- لا تطبق هذا على المستخدمين الجدد بعد هذا التاريخ.

2) منح اشتراك مجاني مفتوح
- أنشئ أو حدّث الاشتراك لهؤلاء المستخدمين ليصبح:
  - plan_type = legacy_free
  - payment_status = waived
  - access_status = active
  - billing_required = false
  - expires_at = null   // اشتراك مفتوح
  - registration_source = legacy_before_payment_update
- يجب أن يتم تجاوز جميع paywalls و checkout لهؤلاء المستخدمين.

3) Backfill / Migration
- نفّذ script أو migration واحدة تقوم بـ:
  - البحث عن كل المستخدمين المسجلين قبل cutoff
  - إنشاء اشتراك مجاني مفتوح لمن لا يملك اشتراكا
  - تحديث الاشتراك الحالي لمن يحتاج تفعيل
  - عدم إنشاء اشتراك مكرر
- يجب أن تكون العملية idempotent:
  أي إذا شُغلت مرتين لا تكرر البيانات ولا تفسد الاشتراكات.

4) التعامل مع المستخدمين الذين لديهم حالات مختلفة
- المستخدم القديم بدون اشتراك: امنحه اشتراكا مفتوحا مجانا
- المستخدم القديم باشتراك منتهي: فعّله كمفتوح مجاني
- المستخدم القديم باشتراك نشط مدفوع: لا تكسر بياناته، لكن اسمح له بالوصول الكامل، ويمكن الإبقاء على سجله المدفوع كما هو مع legacy access
- المستخدم القديم الذي توقف عند بوابة الدفع: امنحه وصولا مباشرا إذا كان حسابه قبل cutoff

5) قاعدة البيانات
إذا احتجت حقولا إضافية فأضفها بشكل نظيف، مثل:
- users.is_legacy_free boolean
- users.legacy_cutoff_group string/null
- subscriptions.plan_type
- subscriptions.access_status
- subscriptions.billing_required
- subscriptions.expires_at
- subscriptions.granted_reason

مثال قيم:
- is_legacy_free = true
- granted_reason = "Registered before payment system update"

6) منطق التطبيق
- عند تسجيل دخول أي مستخدم قديم مشمول:
  - اسمح له بالدخول مباشرة
  - لا ترسله إلى صفحة الدفع
  - أظهر له أن حسابه مفعل
- جميع فحوصات الصلاحية يجب أن تعتبر legacy_free = active access

7) لوحة التحكم
أضف تقريرا بسيطا في admin:
- عدد المستخدمين الذين تم منحهم legacy free access
- تاريخ تنفيذ العملية
- إمكانية تصفية المستخدمين بحسب:
  - legacy free
  - paid
  - sponsored
  - self-funded

8) الأمان
- لا تجعل الواجهة الأمامية تقرر من هو legacy
- القرار يجب أن يكون من backend فقط
- استخدم cutoff ثابت وواضح من السيرفر
- امنع التكرار أو التضارب في الاشتراكات

9) الرسائل
أضف رسالة واضحة للمستخدم القديم بعد تسجيل الدخول:
- Arabic: تم تفعيل اشتراكك المجاني المفتوح لأنك سجلت قبل تحديث نظام الدفع.
- English: Your free open access has been activated because you registered before the payment system update.

ويجب ترجمة هذه الرسالة في جميع لغات النظام عبر i18n، بدون hardcoded text.

10) Acceptance Criteria
- كل مستخدم مسجل قبل cutoff يستطيع الدخول بدون دفع
- لديه وصول كامل ومباشر
- لا يظهر له paywall
- لا يتم إنشاء اشتراكات مكررة
- المستخدمون الجدد فقط هم من يخضعون للنظام الجديد
- يوجد سجل واضح في قاعدة البيانات يبين سبب المنح

ولا تعتمد على تعديل يدوي لكل مستخدم.
أريد حلا جماعيا آليا وآمنا.
